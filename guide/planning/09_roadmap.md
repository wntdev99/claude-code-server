# 1.9 로드맵 가이드

## 목적

프로젝트 일정과 마일스톤을 계획합니다.

---

## 입력

- `result/planning/07_features.md` (백로그)
- `result/planning/08_tech.md` (기술 스택)
- `form/intake_form.md`의 Section 7 (현실적인 것들)
  - 7.1 언제까지 필요한지 (필수)
  - 7.2 제일 먼저 나왔으면 하는 기능
  - 7.3 이미 있는 것들
  - 7.5 개발 참여 인원

## 작업 항목

### 1. 마일스톤 정의

```markdown
## 마일스톤

### M0: 킥오프 (Week 0)
- [x] 기획 완료
- [x] 설계 승인
- [ ] 개발 환경 세팅

### M1: MVP Alpha (Week 1-4)
**목표**: 핵심 기능 동작하는 내부 버전

포함 기능:
- [ ] [핵심 기능 1]
- [ ] [핵심 기능 2]
- [ ] [핵심 기능 3]

완료 기준:
- 내부 테스트 통과
- 주요 버그 없음

### M2: MVP Beta (Week 5-8)
**목표**: 외부 테스터 사용 가능한 버전

포함 기능:
- [ ] Alpha 기능 안정화
- [ ] [추가 기능 1]
- [ ] [추가 기능 2]
- [ ] 사용자 피드백 반영

완료 기준:
- 베타 테스터 10명 사용
- 치명적 버그 없음

### M3: Launch (Week 9-12)
**목표**: 정식 출시

포함 기능:
- [ ] 성능 최적화
- [ ] 결제 연동 (해당시)
- [ ] 스토어 심사 통과

완료 기준:
- 앱스토어 출시
- 모니터링 대시보드 운영
```

### 5. 리스크 및 버퍼

```markdown
## 리스크 및 버퍼

### 일정 리스크

| 리스크 | 영향 | 확률 | 대응 |
|--------|------|------|------|
| 요구사항 변경 | 2주+ | 중간 | Scope 조정 |
| 기술 이슈 | 1주+ | 낮음 | PoC로 사전 검증 |
| 스토어 심사 | 1-2주 | 중간 | 여유 일정 확보 |
| 팀원 이탈 | 2주+ | 낮음 | 문서화, 페어링 |

### 버퍼 계획

- 각 마일스톤에 20% 버퍼 포함
- 전체 일정에 2주 버퍼 확보
- 최소 기능으로 일정 준수 vs 기능 추가로 일정 지연 시 → 일정 준수 우선
```

```

gantt
    title 프로젝트 타임라인
    dateFormat  YYYY-MM-DD
    section 기획
    기획 완료           :done, p1, 2025-01-01, 2w
    section 설계
    설계                :active, p2, after p1, 2w
    section 개발
    MVP Alpha           :p3, after p2, 4w
    MVP Beta            :p4, after p3, 4w
    section 출시
    최적화 및 출시      :p5, after p4, 2w
\`\`\`
```
