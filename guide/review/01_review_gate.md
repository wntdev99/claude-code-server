# 검토 게이트 가이드

## 개요

프로젝트 진행 중 사람의 검토와 승인이 필요한 시점(검토 게이트)과 검토 방법을 안내합니다.

---

## 2. GATE 1: 기획 검토

### 2.1 검토 시점

기획 단계(1.1 ~ 1.9) 완료 후, 설계 단계 진입 전

### 2.2 검토 대상 문서

```
┌─────────────────────────────────────────────────────────────┐
│ 기획 검토 대상                                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📁 확인 위치: result/planning/                             │
│                                                             │
│  필수 검토 문서:                                            │
│  ───────────────                                            │
│  ☐ 01_idea.md          아이디어 정의                        │
│  ☐ 02_market.md        시장 분석                            │
│  ☐ 03_persona.md       페르소나                             │
│  ☐ 04_user_journey.md  사용자 여정                          │
│  ☐ 05_business_model.md 비즈니스 모델                       │
│  ☐ 06_product.md       제품 정의                            │
│  ☐ 07_features.md      기능 정의                            │
│  ☐ 08_tech.md          기술 검토                            │
│  ☐ 09_roadmap.md       로드맵                               │
│                                                             │
│  Figma 확인 (있는 경우):                                    │
│  ─────────────────────                                      │
│  ☐ 01_Planning > Personas                                   │
│  ☐ 01_Planning > User Journey                               │
│  ☐ 01_Planning > Information Architecture                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 검토 체크리스트

```markdown
## 기획 검토 체크리스트

### 아이디어 (01_idea.md)
- [ ] 문제 정의가 명확한가?
- [ ] 솔루션이 문제를 해결하는가?
- [ ] 성공 기준이 측정 가능한가?

### 시장 분석 (02_market.md)
- [ ] TAM/SAM/SOM이 합리적인가?
- [ ] 경쟁사 분석이 충분한가?
- [ ] 차별화 포인트가 명확한가?

### 사용자 분석 (03_persona.md, 04_user_journey.md)
- [ ] 타겟 사용자가 명확한가?
- [ ] 페르소나가 현실적인가?
- [ ] Pain Point와 Needs가 구체적인가?
- [ ] 사용자 여정이 논리적인가?

### 비즈니스 모델 (05_business_model.md)
- [ ] 수익 모델이 실현 가능한가?
- [ ] 가격 정책이 합리적인가?
- [ ] 재무 예측이 현실적인가?

### 제품 정의 (06_product.md)
- [ ] MVP 범위가 적절한가?
- [ ] KPI가 측정 가능한가?
- [ ] Out of Scope가 명확한가?

### 기능 정의 (07_features.md)
- [ ] User Story가 명확한가?
- [ ] 우선순위가 합리적인가?
- [ ] Acceptance Criteria가 구체적인가?

### 기술 검토 (08_tech.md)
- [ ] 기술 스택이 적절한가?
- [ ] 기술 리스크가 관리 가능한가?
- [ ] PoC 필요 항목이 식별되었는가?

### 로드맵 (09_roadmap.md)
- [ ] 마일스톤이 현실적인가?
- [ ] 리소스 계획이 적절한가?
- [ ] 리스크 버퍼가 있는가?
```

### 2.4 검토 결과 처리

```
┌─────────────────────────────────────────────────────────────┐
│ 검토 결과 처리                                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 승인 (Approved)                                         │
│  ─────────────────                                          │
│  → status.md에 "기획 검토: ✅ 승인" 기록                    │
│  → Phase 2: 설계 단계 진행                                  │
│                                                             │
│  ⚠️ 조건부 승인 (Approved with Comments)                    │
│  ─────────────────────────────────────────                  │
│  → 지적 사항 반영                                           │
│  → 수정된 문서 업데이트                                     │
│  → Phase 2 진행 (수정 병행)                                 │
│                                                             │
│  ❌ 반려 (Rejected)                                         │
│  ─────────────────                                          │
│  → 피드백 내용 확인                                         │
│  → 해당 문서 수정                                           │
│  → 재검토 요청                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 3. GATE 2: 설계 검토

### 3.1 검토 시점

설계 단계(2.1 ~ 2.5) 완료 후, 개발 단계 진입 전

### 3.2 검토 대상 문서

```
┌─────────────────────────────────────────────────────────────┐
│ 설계 검토 대상                                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📁 확인 위치: result/design/                               │
│                                                             │
│  필수 검토 문서:                                            │
│  ───────────────                                            │
│  ☐ 01_screen.md        화면 설계                            │
│  ☐ 02_data_model.md    데이터 모델                          │
│  ☐ 03_task_flow.md     태스크 플로우                        │
│  ☐ 04_api.md           API 설계                             │
│  ☐ 05_architecture.md  아키텍처                             │
│                                                             │
│  Figma 확인 (필수):                                         │
│  ─────────────────                                          │
│  ☐ 02_Design > Wireframes                                   │
│  ☐ 02_Design > UI Design                                    │
│  ☐ 02_Design > Prototypes                                   │
│  ☐ 03_Components > Design System                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 검토 체크리스트

```markdown
## 설계 검토 체크리스트

### 화면 설계 (01_screen.md + Figma)
- [ ] 모든 화면이 정의되었는가?
- [ ] 화면 흐름이 논리적인가?
- [ ] 상태별 UI가 정의되었는가? (Default, Loading, Empty, Error)
- [ ] 와이어프레임/UI가 기능 요구사항을 충족하는가?
- [ ] 프로토타입 인터랙션이 자연스러운가?

### 데이터 모델 (02_data_model.md)
- [ ] 엔티티 정의가 명확한가?
- [ ] 관계가 올바르게 설정되었는가?
- [ ] 필수/선택 필드가 구분되었는가?
- [ ] 인덱스 설계가 적절한가?

### 태스크 플로우 (03_task_flow.md)
- [ ] 주요 플로우가 모두 정의되었는가?
- [ ] 예외 처리가 고려되었는가?
- [ ] 에러 시나리오가 정의되었는가?

### API 설계 (04_api.md)
- [ ] 엔드포인트가 RESTful한가?
- [ ] 요청/응답 형식이 명확한가?
- [ ] 에러 응답이 정의되었는가?
- [ ] 인증/권한이 고려되었는가?

### 아키텍처 (05_architecture.md)
- [ ] 시스템 구성이 명확한가?
- [ ] 확장성이 고려되었는가?
- [ ] 보안이 고려되었는가?
- [ ] 모니터링/로깅이 계획되었는가?
```

### 3.4 검토 결과 처리

기획 검토와 동일한 프로세스:
- ✅ 승인 → Phase 3: 개발 진행
- ⚠️ 조건부 승인 → 수정 후 개발 진행
- ❌ 반려 → 수정 후 재검토

## 4. 검토 요청 방법

### 4.1 검토 요청 메시지 템플릿

```markdown
## 🚦 [Phase 1/2] 검토 요청

### 검토 대상
[기획/설계] 단계 산출물 검토를 요청드립니다.

### 확인 위치

#### 문서
| 문서 | 링크 | 상태 |
|-----|------|------|
| 아이디어 정의 | [01_idea.md](result/planning/01_idea.md) | ✅ 완료 |
| 시장 분석 | [02_market.md](result/planning/02_market.md) | ✅ 완료 |
| ... | ... | ... |

#### Figma
| 항목 | 링크 |
|-----|------|
| 기획 시각화 | [01_Planning](https://figma.com/file/xxx) |
| UI 디자인 | [02_Design](https://figma.com/file/xxx) |
| 프로토타입 | [Prototype](https://figma.com/proto/xxx) |

### 검토 방법
1. 위 문서 링크를 순서대로 검토
2. Figma 링크에서 시각 자료 확인
3. 피드백은 아래 양식으로 작성

### 피드백 양식
```
문서명:
피드백 유형: [수정 필요 / 질문 / 제안]
내용:
```

### 검토 기한
YYYY-MM-DD까지 검토 부탁드립니다.

### 결정 사항
- [ ] ✅ 승인 (다음 단계 진행)
- [ ] ⚠️ 조건부 승인 (피드백 반영 필요)
- [ ] ❌ 반려 (재검토 필요)
```

### 4.2 status.md 업데이트

검토 요청 시 status.md 업데이트:

```markdown
## 현재 상태: Phase 1 - 기획 검토 대기

### 완료된 단계
- [x] 1.1 아이디어 정의
- [x] 1.2 시장 분석
- [x] 1.3 페르소나
- [x] 1.4 사용자 여정
- [x] 1.5 비즈니스 모델
- [x] 1.6 제품 정의
- [x] 1.7 기능 정의
- [x] 1.8 기술 검토
- [x] 1.9 로드맵

### 🚦 대기 중인 검토
- **기획 검토** (요청일: YYYY-MM-DD)
  - 검토자: [이름]
  - 기한: YYYY-MM-DD
  - 상태: ⏳ 검토 중

### 다음 작업
- 기획 검토 승인 시 → Phase 2: 설계 진행
```

## 5. 피드백 처리

### 5.1 피드백 수렴

```
┌─────────────────────────────────────────────────────────────┐
│ 피드백 처리 프로세스                                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 피드백 수집                                             │
│     • 문서/Figma에 대한 코멘트                              │
│     • 회의록/슬랙 메시지                                    │
│                                                             │
│  2. 피드백 분류                                             │
│     ┌───────────────────────────────────────────┐          │
│     │ 유형        │ 처리                        │          │
│     ├─────────────┼─────────────────────────────┤          │
│     │ 필수 수정   │ 즉시 반영                   │          │
│     │ 권장 수정   │ 가능하면 반영               │          │
│     │ 질문       │ 답변 후 필요시 반영          │          │
│     │ 제안       │ 검토 후 다음 버전에 반영     │          │
│     └───────────────────────────────────────────┘          │
│                                                             │
│  3. 피드백 반영                                             │
│     • 해당 문서 수정                                        │
│     • 변경 사항 기록 (버전 히스토리)                        │
│                                                             │
│  4. 재확인 요청                                             │
│     • 수정 완료 알림                                        │
│     • 필요시 재검토 요청                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 피드백 추적

```markdown
## 피드백 추적 로그

| ID | 문서 | 피드백 내용 | 유형 | 상태 | 처리일 |
|----|-----|------------|------|------|-------|
| F001 | 01_idea.md | 성공 기준 구체화 필요 | 필수 | ✅ 완료 | 01/15 |
| F002 | 03_persona.md | Secondary 페르소나 추가 | 권장 | ✅ 완료 | 01/16 |
| F003 | 07_features.md | P0 기능 재검토 | 질문 | ✅ 완료 | 01/16 |
```

## 체크리스트

### 검토 요청 전
```
☐ 모든 단계 문서 작성 완료
☐ Figma 시각화 작업 완료 (해당 시)
☐ 문서 내 Figma 링크 추가
☐ status.md 업데이트
☐ 검토 요청 메시지 작성
```

### 검토 중
```
☐ 검토자 질문에 응답
☐ Figma 코멘트 확인
☐ 피드백 수집 및 분류
```

### 검토 후
```
☐ 피드백 반영
☐ 수정 문서 업데이트
☐ 승인 결과 status.md 기록
☐ 다음 단계 준비
```
